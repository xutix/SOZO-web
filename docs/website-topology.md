## 3. 用户理解路径

官网不应该让用户一次性读完所有业务，而是按“认识品牌 → 理解产品 → 查看能力 → 建立合作”的路径逐步进入。

```mermaid
flowchart LR
    A["品牌官网"] --> B["认识 SOZO"]
    A --> C["理解产品方向"]
    A --> D["重点看 SOZO Dock"]
    A --> E["看 FPV / 无人机能力"]
    A --> F["看案例与图集"]
    A --> G["联系合作"]

    B --> H["关于我们"]
    C --> I["产品方向页"]

    E --> K["FPV 页面"]

    G --> M["页脚 / 联系入口"]
```

## 4. 后续扩展建议

SOZO Dock 发布后，官网需要从“品牌展示网站”扩展为“产品官网 + 用户支持中心”。建议提前预留以下页面和信息结构。

```mermaid
flowchart TD
    A["SOZO-web 官网"] --> B["品牌与产品展示"]
    A --> C["SOZO Dock 产品生态"]
    A --> D["支持中心 Support"]
    A --> E["下载中心 Download"]
    A --> F["更新日志 Changelog"]

    C --> C1["产品介绍"]
    C --> C2["功能说明"]
    C --> C3["软件界面演示"]
    C --> C4["真实用户场景"]

    D --> D1["快速开始"]
    D --> D2["用户手册"]
    D --> D3["常见问题"]
    D --> D4["故障排查"]

    E --> E1["Windows App"]
    E --> E2["macOS App"]
    E --> E3["历史版本"]

    F --> F1["软件更新"]
    F --> F2["固件能力说明"]
    F --> F3["已知限制"]
```

建议后续新增但不急于一次性完成的页面：

- `support.html`：用户手册、快速开始、常见问题。
- `download.html`：Windows / macOS 桌面软件下载入口。
- `changelog.html`：软件更新记录、功能状态说明。

## 5. 仓库边界提醒

SOZO-web 是官网展示层，不实现真实硬件控制。

官网可以展示：

- 产品定位
- 功能说明
- 软件界面演示
- 下载入口
- 用户文档
- 产品路线图

官网不应该实现：

- ESP32 固件逻辑
- Electron 主进程硬件控制
- 串口通信执行
- BLE HID 执行
- 系统音频采集
- 固件升级真实逻辑

后续涉及 SOZO Dock 真实能力时，应以 SOZO-DOCK 仓库的设备能力文档为准，避免把计划中、部分支持或未完成能力写成已经完整发布。
